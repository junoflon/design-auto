# Phase 분리 계획

> 각 Phase 완료 기준 = **실제 Railway에 배포 + 팀원 1명이 실업무에 써봄 + 피드백 반영**

---

## Phase 1 — 편집·품질 MVP (우선순위 1)

**목표**: "생성된 결과를 버리지 않고 다듬어서 쓸 수 있다"

### 범위
- [ ] SQLite 도입 (`better-sqlite3`), `Project/Asset/AssetVersion` 3개 테이블
- [ ] 생성 결과를 DB에 저장, 새로고침해도 유지
- [ ] **인라인 텍스트 편집** — contenteditable + 저장 시 HTML 스냅샷
- [ ] **간단 스타일 인스펙터** — 선택 요소의 색·크기·폰트 조정
- [ ] **이미지 영역 교체** — 업로드 / URL 붙여넣기
- [ ] **블록 단위 재생성** — "이 슬라이드만 다시" 버튼
- [ ] **Undo/Redo + 버전 스냅샷** (AssetVersion 자동 적재)
- [ ] **미리보기 → 확정** 플로우 (확정 전엔 초안 상태)
- [ ] **Claude 응답 파싱 실패 자동 1회 재시도**

### 장점/단점
- 장점: 실무자가 재생성 루프 없이 바로 쓸 수 있는 최소 수준
- 단점: 브랜드 프리셋 아직 수동 입력, 팀 인증 없음

### 완료 기준
실무자 1명이 썸네일 5개 생성 → 텍스트·색 수정 → 저장 → 다운로드까지 문제없이 수행

---

## Phase 2 — 브랜드·워크플로우 (우선순위 2)

**목표**: "반복 작업이 빨라진다"

### 범위
- [ ] **BrandPreset CRUD** — 로고/컬러/폰트/톤 메모 저장
- [ ] 생성 시 프리셋 선택 → 프롬프트에 자동 주입
- [ ] **프로젝트(Project)** 개념 도입 — 좌측 사이드바 네비게이션
- [ ] 프로젝트별 에셋 리스트 (썸네일·필터·검색)
- [ ] **복제·즐겨찾기**
- [ ] **레퍼런스 이미지 업로드 → Claude Vision으로 스타일 추출** → 프롬프트 자동 보강
- [ ] 현재 `analyze-reference` 엔드포인트 강화: 색 팔레트·레이아웃·톤 JSON 요약

### 장점/단점
- 장점: 같은 캠페인 안에서 결과물 톤 일관성 확보, 반복 작업 시간 단축
- 단점: Phase 1 안정화 필수. 레퍼런스 비전 호출로 API 비용 증가

### 완료 기준
브랜드 프리셋 2개 등록 → 각 프리셋으로 카드뉴스 5장씩 생성 → 톤 일관성 육안 확인

---

## Phase 3 — 운영·안정성 (우선순위 3)

**목표**: "팀이 안심하고 쓴다"

### 범위
- [ ] **Basic Auth** 미들웨어 — 단일 팀 비밀번호 (환경변수 `TEAM_PASSWORD`)
- [ ] **User 간이 구분** — 로그인 시 이름 입력, 쿠키 저장 (작성자 표시용)
- [ ] **UsageLog 적재** — 모든 생성/재생성 요청에 토큰·비용 기록
- [ ] **사용량 대시보드** — 이번 달 토큰·USD, 사용자별 랭킹
- [ ] **월 한도 설정** — 초과 시 경고 배너 (차단까지는 아님)
- [ ] **에러 로그 뷰어** — 실패 요청 원인 열람 (운영자용)
- [ ] **Railway 배포 가드** — 헬스체크 강화, DB 볼륨 마운트, 자동 재시작
- [ ] **백업 스크립트** — SQLite 파일 일일 스냅샷

### 장점/단점
- 장점: 비용 블랙홀 방지, 실패 디버깅 가능, 팀 공유 안전
- 단점: 본 기능 개발은 아니라 미룰 수 있지만, 한 번 사고 나면 복구 비용 큼

### 완료 기준
1달간 팀원 3명이 실사용 → 비용 이상 없음, 데이터 손실 없음, 평균 응답 시간 60초 이내

---

## Out of Scope (v1 전체)

- 결제/구독/공개 판매
- 동시 편집(멀티 커서)
- 공개 공유 링크
- 벡터 출력(SVG)
- 모바일 전용 UI
- 외부 이미지 API(Unsplash, Pexels) 자동 연동 — v2 후보

## Phase 간 의존성

```
Phase 1 (편집·DB)
   ↓ 필수
Phase 2 (프리셋·프로젝트는 DB 스키마 확장)
   ↓ 권장
Phase 3 (인증·로그는 독립적이지만 DB 있어야 의미 있음)
```

Phase 1 완성 전에는 Phase 2·3 손대지 말 것. DB 스키마가 바뀌면 전부 다시 마이그레이션.
